This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Project Is

**SkillPack** aggregates AI agent skills from skills.sh and GitHub, categorizes them by developer role, and serves them as curated bundles with a single install command per platform (Claude Code, Cursor, Copilot, Continue.dev, Universal).

- ~1,500 Tier 1 skills stored fully in PostgreSQL (quality_score ≥ 5)
- ~110k Tier 2 skills indexed by reference, live-fetched from GitHub on demand
- 62+ curated bundles (role-based + task-based) generated by the pipeline

## Architecture

### Data Flow

```
skills.sh scraper ─┐
                   ├─→ raw skills → tagger.py (Groq LLM) → ingestion.py → PostgreSQL
GitHub crawler ────┘                                                           │
                                                                               ↓
                                    bundle_generator.py (keyword pre-filter → Groq AI curation)
                                                        → Bundle + BundleCommand rows
                                                                               │
                                                                               ↓
                                                                    FastAPI → Next.js frontend
```

### Tier 1 vs Tier 2

| | Tier 1 | Tier 2 |
|---|---|---|
| Storage | Full row in `skills` table | Lightweight row in `skills_index` |
| Content | `raw_content` stored in DB | Fetched live from GitHub on demand |
| Threshold | `quality_score >= 5` + `install_count >= TIER1_MIN_INSTALLS` (default 50) | Everything else |
| API | `/api/skills/{slug}` | `/api/live/{owner}/{repo}` |

Tier 2 live-fetches are Redis-cached for 24 hours.

### Key Data Models (`db/models.py`)

**`Skill`** - The core entity (Tier 1 only):
- `slug`: `owner/repo` or `owner/repo/skillId` (3-part for skills.sh)
- `primary_category`: one of `frontend, backend, fullstack, devops, ml-ai, security, database, testing, cloud, mobile, data-science`
- `platforms`: array, default `["claude_code"]`
- `quality_score`: 0–10 float, computed by Groq tagger
- `content_hash`: MD5 of `raw_content` - used to skip re-processing unchanged skills

**`Bundle`** → `BundleCommand` (one row per platform):
- `bundle.skill_ids`: ordered list of Skill IDs chosen by AI curation
- `bundle.commands?.[platform]` on the frontend = `bundle_commands.command` for that platform
- Commands generated by `pipeline/install_generator.py` as multi-slug npx commands

**`CrawlJob`** - Audit trail for every crawl execution.

### API Routes (`api/routes/`)

| Prefix | File | Notes |
|---|---|---|
| `/api/bundles` | `bundles.py` | List/get bundles. Redis-cached 5 min. `install/{platform}` increments `install_count` (not cached). |
| `/api/skills` | `skills.py` | List with filters, categories, get by `/{slug:path}` |
| `/api/search` | `search.py` | Full-text (`tsvector`), fuzzy (`ILIKE`), AI bundle match (Groq) |
| `/api/live` | `live.py` | Live-fetch Tier 2 skills, Redis cache 24hr |
| `/api/crawl` | `crawl.py` | Admin-only (`X-Admin-Token` header), triggers Celery tasks |
| `/api/cron` | `cron.py` | Vercel cron endpoints - skill discovery (daily 1am) + bundle curator (daily 3am) |

`redirect_slashes=False` is set on the FastAPI app - all router list endpoints use `@router.get("")` (not `"/"`).

### Frontend Routes (`frontend/src/app/`)

| Route | Key behavior |
|---|---|
| `/` | Rotating placeholder search → `/api/search/match-bundle` → redirects to best bundle |
| `/explore` | Tabs: bundles (type filter) / skills (live search). AnimatePresence tab transitions. |
| `/bundle/[slug]` | Platform selector drives which `commands[platform]` is shown. Skills expand in-place. |
| `/skills/[...slug]` | Catch-all for `owner/repo/skillId` 3-part slugs. Quality score bar, raw SKILL.md toggle. |

`frontend/src/lib/api.ts` is the single typed client. All pages are `"use client"` + `useEffect` for data fetching - `next: { revalidate }` has no effect here. A 2-minute module-level in-memory cache (`_mem` Map) is applied to all `get()` calls except `installCommand` (which has side effects).

### Pipeline (`pipeline/`)

- **`tagger.py`**: Calls Groq (`llama-3.3-70b-versatile`) with batches of skills. Falls back to keyword heuristics if LLM fails or rate-limited.
- **`bundle_generator.py`**: Two-phase curation:
  1. Keyword pre-filter - SQL query matching skill name/description/tags/role_keywords against bundle's keyword list
  2. Groq AI filter - sends up to 60 candidates to Groq, which returns only truly relevant slugs ordered by relevance. No fallback padding - if keywords don't match, bundle stays small.
- **`install_generator.py`**: Builds platform-specific install commands.

### Cron Jobs (`backend/vercel.json`)

Two daily Vercel crons (Hobby tier max = once/day):
- `0 1 * * *` → `/api/cron/skill-discovery`: fetches new GitHub skills, deduplicates by slug, tags and ingests
- `0 3 * * *` → `/api/cron/bundle-curator`: AI re-curates the 10 least-recently-updated bundles via Groq

| `TIER1_MAX_SKILLS` | - | `25000` | Cap on Tier 1 DB size |

## Important Patterns

**Supabase Session Pooler required**: asyncpg uses server-side prepared statements which conflict with Supabase's Transaction Pooler (PgBouncer, port 6543). Always use the **Session Pooler** (`aws-1-ap-southeast-1.pooler.supabase.com:5432`). NullPool is used on both engines for Vercel serverless compatibility.

**Bulk ingestion**: `db/ingestion.py` uses `bulk_insert_mappings` + batch `IN` queries. Do not revert to per-row `db.commit()` - it causes 300s+ timeouts on Supabase.

**3-part skill slugs**: skills.sh skills have slugs like `owner/repo/skillId`. The API route uses `/{slug:path}`, and the frontend uses `[...slug]` to reassemble with `.join("/")`.

**Content dedup**: `ingestion.py` computes MD5 hash of `raw_content` and skips re-tagging if `content_hash` matches existing row.

**Framer Motion + async data**: Do not use `whileInView` with `once: true` on a parent grid that renders skeleton placeholders while data loads. When bundles load and React remounts children with new keys, the parent's `whileInView` has already fired and children stay at `opacity: 0`. Instead, put `whileInView` directly on each child card so each independently animates when it mounts.

**Bundle curation with Groq**: `bundle_generator._ai_filter_skills()` sends candidates as `slug | name | tags` lines and parses the response with `re.search(r'\[.*?\]', raw, re.DOTALL)` - robust against Groq wrapping the array in markdown or explanation text.

**Redis cache invalidation**: Bundle list/detail caches (`bundles:list:*`, `bundles:slug:*`) use 5-minute TTL. They are not explicitly invalidated on write - cron jobs and pipeline runs will reflect after TTL expiry.
